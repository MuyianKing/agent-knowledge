# HTML → Vue 组件化转换

将一个静态 HTML 页面（含 `<style>` / `<script>`）转换为符合本仓库技术栈的 Vue 组件化页面。

> **技术栈、命名、代码风格、组件库选型等所有项目级约束，一律以仓库内的项目规则文件为准**（例如 `.trae/rules/project_rules.md`、`CLAUDE.md`、`.eslintrc*`、`.prettierrc`、`jsconfig.json`、`package.json` 中体现的依赖与脚本）。
> 本 skill **只关注"如何转换"的方法论**，不重复也不锁死任何具体技术栈。

## 触发场景

- 用户说"把 xxx.html 转成 vue 页面/组件"
- 用户给出静态原型 HTML 想要落地为 Vue 实现
- 用户提到"组件化"、"拆组件" + 已有静态 HTML

## 必问前置

**在开始任何文件读写前**，使用 `AskUserQuestion` 工具确认以下信息（用户已在消息中明确提供则跳过对应项）：

1. **源 HTML 文件**的绝对路径或工作区相对路径
2. **页面主入口 Vue 文件**的目标位置

> **子组件目录默认就是主入口同级的 `components/` 目录，不要询问。**
> 例如主入口为 `src/views/link-device/Index.vue`，子组件统一放 `src/views/link-device/components/`。

## 开工前阅读项目规则

在动手转换前，**主动读取并遵循**以下文件中的项目级约束（如果存在）：

- `.trae/rules/project_rules.md`
- `CLAUDE.md` / `AGENTS.md` / 仓库根的 `README.md`
- `.eslintrc*` / `.prettierrc` / `.editorconfig`
- `jsconfig.json` / `tsconfig.json`
- `package.json`（依赖、scripts、engines）
- `vue.config.js` / `vite.config.*`（路径别名、插件、auto-import 等）

> 任何与本 skill 描述冲突的项目规则，**以项目规则为准**。

## 转换原则（核心方法论）

> 一句话：**按项目栈做事，但视觉与语义说了算；少即是多，留好接口就行。**

1. **够用就好，不做多余的事**
   - HTML 中存在但**未在 DOM 渲染**的功能（如 JS 里有但页面没用到的 stats、download 等）**不要转**
   - 不为"以后可能要"提前抽象、不加无用 try/catch、不加无用类型校验
   - 不主动写 README / docs / 测试

2. **拥抱项目既有技术栈**
   - 表格、表单、弹窗、Toast、图标等，优先选用项目已经在用的组件库 / 工具，不引入新依赖
   - 具体用什么版本、什么 API，**以项目规则与现有代码为准**

3. **拥抱归拥抱，但不教条**
   - "保持原样式不变" > "用组件库默认样式"
   - 必要时通过样式深选择器覆盖组件库内部类还原视觉（深选语法以项目规则为准）
   - **图标语义不匹配时保留原 SVG**（例如品牌图标、停止方块等独特造型），不强行套组件库图标
   - 视觉 / 语义一致性 > 框架原生感

4. **关注点分离 / 组件化分层**
   ```
   <模块>/
     Index.vue              # 页面主入口：组合 + 生命周期 + 事件桥接
     components/
       PageHeader.vue       # 头部
       <主功能>.vue          # 列表/表格/卡片等大块
       <Badge>.vue / ...    # 高复用细粒度展示组件
       use<Domain>.js       # composable：数据 + 操作 + 接口替换点
   ```
   组件命名、文件后缀等格式细节以项目规则为准。

5. **数据要"对得上未来的接口"**
   - mock 数据放在 composable / store / service 层（按项目惯例），结构对齐预期后端字段
   - 用项目支持的响应式 API 持有列表，方便整体替换
   - 提前留 `loading` 等状态位
   - 函数命名 `fetchXxx` / `startXxx` / `stopXxx` / `refreshXxx`，TODO 注释标注接口替换点

6. **布局要可控**
   - 内容溢出场景：固定头部 / 工具栏，**只让列表区滚动**
   - flex 链路上每一级 `min-height: 0`，避免被内容撑开
   - 表格固定表头时优先用组件库自带能力（如 `height` / sticky 头）

7. **CSS 处理**
   - 把原 `:root { --xxx }` 变量挪到**页面根类**作用域下，**避免污染全局**
   - 删除原 HTML 中与预览器注入、外链字体、第三方分析脚本等无用的样式 / 标签
   - 媒体查询、动画 `@keyframes` 保留
   - 样式语法（scoped、lang、深选）以项目规则为准

## 推荐流程（执行步骤）

1. **询问缺失信息**（源 HTML 路径 + 主入口位置；子组件目录默认 `components/` 不问）
2. **读取项目规则文件**（见上文"开工前阅读项目规则"）
3. **创建 TodoWrite 计划**：①子组件 ②数据/逻辑层 ③主入口 ④lint
4. **读源 HTML**，盘点：
   - 哪些 DOM 块可拆成独立组件（头部、工具栏、列表项、Badge…）
   - 哪些 JS 函数实际被 DOM 使用 → 保留；未使用 → 删除
   - 哪些 SVG 可换组件库图标而**语义不偏离**
5. **先建子组件 + 数据/逻辑层**，再写主入口
6. **样式还原**：保留原配色变量，按项目规则使用深选覆盖组件库默认主题
7. **最后**按项目规则跑 lint / typecheck（命令以 `package.json` scripts 为准）
8. 简要总结改动（视觉是否变化 / 接口对接点 / 滚动方案 等）

## 约束清单（最终自检）

- [ ] 没用到的 HTML 块 / JS 函数都未被转换
- [ ] 严格遵循了项目规则文件中的所有约束（技术栈、命名、风格、深选语法等）
- [ ] 没有引入项目未使用的新依赖
- [ ] 数据层留好了接口替换点，TODO 注释清晰
- [ ] 视觉效果与原 HTML 一致（除非用户明确要求变更）
- [ ] lint / typecheck（如适用）通过

## 反面示例（禁止）

- ❌ 用户没说源 HTML 或主入口位置就直接动手
- ❌ 反过来：用户已给出主入口，还询问子组件目录（默认就在主入口同级 `components/`）
- ❌ 把未渲染的 DOM / 未使用的 JS 也照搬转换
- ❌ 不读项目规则就凭经验脑补技术栈
- ❌ 引入项目未使用的新组件库 / 状态管理 / 构建工具
- ❌ 把品牌专属 SVG 强行换成组件库图标
- ❌ 主动加单元测试 / 文档 / README
